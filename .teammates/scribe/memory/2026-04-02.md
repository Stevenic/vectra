## Protobuf Feasibility Analysis

Researched the current on-disk storage format to assess the scope of adding protocol buffer support. Analyzed all serialization touchpoints in LocalIndex.ts, LocalDocumentIndex.ts, and the FileStorage abstraction layer.

### Key findings
- All storage is JSON via hardcoded `JSON.stringify()`/`JSON.parse()` тАФ no serialization abstraction exists
- ~10 serialization touchpoints across 2 main files
- FileStorage interface is well-abstracted (good), but the format layer above it is not
- Vectors stored as inline JSON number arrays (~6KB per 1536-dim vector in text vs ~6KB in binary тАФ the binary savings come mainly from eliminating JSON overhead like brackets, commas, and text-encoded floats)

### Files reviewed
- `src/LocalIndex.ts`, `src/LocalDocumentIndex.ts`, `src/storage/FileStorage.ts`

## gRPC Server & Language Bindings Spec

Wrote the full architecture spec for the gRPC server and language bindings at `specs/grpc-server-and-bindings.md`.

### Design decisions (from stevenic)
1. **Single-index vs multi-index:** Support both тАФ `vectra serve <index>` and `vectra serve --root <dir>`
2. **Embeddings:** Server-side тАФ clients send text, server computes vectors
3. **Process lifecycle:** Support both foreground and daemon (`--daemon` flag + `vectra stop`)
4. **Auth:** Localhost only for v1 (bind `127.0.0.1`, no auth tokens)
5. **Languages:** Python, C#, Rust

### What the spec covers
- Full proto service definition (`VectraService`) with 18 RPCs covering index management, item CRUD, queries, document operations, stats, and lifecycle
- CLI additions: `vectra serve` (with `--root`, `--port`, `--daemon`, `--pid-file`) and `vectra stop`
- Server architecture: `VectraServer`, `IndexManager`, handler modules
- New dependencies: `@grpc/grpc-js`, `@grpc/proto-loader` (required, not optional)
- `bindings/` directory structure with Python (`vectra-client`), C# (`Vectra.Client`), Rust (`vectra-client` crate)
- Idiomatic wrapper API examples for all three languages
- 5-phase rollout: proto+server тЖТ Python тЖТ C# тЖТ Rust тЖТ docs/samples
- 4 open questions (stub generation strategy, package publishing, hot-reload, connection draining)

## Protobuf Feasibility Analysis

Researched the current on-disk storage format to assess the scope of adding protocol buffer support. Analyzed all serialization touchpoints in LocalIndex.ts, LocalDocumentIndex.ts, and the FileStorage abstraction layer.

### Key findings
- All storage is JSON via hardcoded `JSON.stringify()`/`JSON.parse()` — no serialization abstraction exists
- ~10 serialization touchpoints across 2 main files
- FileStorage interface is well-abstracted (good), but the format layer above it is not
- Vectors stored as inline JSON number arrays (~6KB per 1536-dim vector in text vs ~6KB in binary — the binary savings come mainly from eliminating JSON overhead like brackets, commas, and text-encoded floats)

### Files reviewed
- `src/LocalIndex.ts`, `src/LocalDocumentIndex.ts`, `src/storage/FileStorage.ts`

## Spec Update — Resolved Open Questions

Updated `specs/binary-storage-format.md` to resolve all four open questions based on stevenic's decisions:

1. **protobufjs (~3 MB unpacked) → optional dependency** — runtime check with clear error message if not installed
2. **`--format` flag on `vectra create`, format inferred thereafter** — all subsequent commands auto-detect from file extension on disk
3. **float32 for vectors, float64 for norms** — ~50% size savings, negligible precision impact (cosine similarity error ~10⁻⁷), matches industry practice
4. **`migrate()` function in both API and CLI** — no mixed-format indexes, explicit `migrateIndex()` API and `vectra migrate` CLI command with interruption recovery
